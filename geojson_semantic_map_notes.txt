PROJECT NOTES — GEOJSON / SEMANTIC MAP / GNSS / HEIGHT
Generated on 2026-03-13

1. MAIN GEOJSON DECISION
- Treat the GeoJSON as the mapped semantic result, not as the raw image annotation.
- Keep raw detections (RGB bbox, mask, depth, camera info, timestamps, TF) in the rosbag / ROS outputs.
- Use GeoJSON only for georeferenced objects that are already transformed into map/global coordinates.

2. GEOMETRY TYPE POLICY
Recommended default policy:
- Segmentation detections:
  - Prefer Polygon or MultiPolygon.
  - Build the mapped geometry from the segmentation mask, not from the image bbox.
- Bounding box detections:
  - Do NOT treat the raw image bbox as a real geographic polygon.
  - Use a bbox-derived footprint polygon only if depth + projection make it trustworthy.
  - Otherwise keep bbox as metadata (pixel_bbox) and optionally fall back to Point or clearly marked proxy polygon.

3. BIGGEST CHANGE TO MAKE
- Stop using bbox width/height + minimum extent as the main semantic-map footprint when a segmentation mask exists.
- Instead:
  1) extract contour from the mask,
  2) back-project contour or sampled mask pixels with depth into 3D,
  3) transform to world/global frame,
  4) flatten/project into the map surface representation you want,
  5) simplify the outline,
  6) export as Polygon / MultiPolygon.

4. DO NOT CONFUSE IMAGE SPACE WITH MAP SPACE
- Image bbox = detector result in pixel coordinates.
- GeoJSON geometry = real mapped shape in lon/lat.
- Keep the original detector bbox in properties as pixel_bbox, not as GeoJSON bbox and not as the main map footprint.

5. COORDINATE ORDER
- GeoJSON positions should use [longitude, latitude] as the first 2 values.
- Keep this consistent everywhere.

6. WHAT THE 3RD COORDINATE CURRENTLY IS
- The current 3rd coordinate is coming from GNSS, so it is acting like altitude/elevation of the mapped point.
- It is NOT object height.
- If it is raw GNSS height, it is probably ellipsoidal height, not true "height above sea level".
- Do not describe it as sea-level altitude unless you verify a geoid correction is applied.

7. GNSS HEIGHT VS SEA LEVEL
- Raw GNSS usually gives ellipsoidal height.
- "Height above sea level" usually refers to orthometric height / elevation.
- These are not the same.
- They can differ by tens of meters depending on location.
Action:
- Check whether your pipeline applies a geoid model / altitude conversion.
- If no correction is applied, rename the field more carefully, e.g. ellipsoidal_height_m or gnss_altitude_m.

8. SHOULD GEOMETRY INCLUDE THE 3RD COORDINATE?
Recommended default:
- Use 2D GeoJSON geometry: [lon, lat]
- Store vertical information in properties.

Suggested properties:
- base_altitude_m (or gnss_altitude_m / ellipsoidal_height_m if that is what it really is)
- top_altitude_m (if available)
- height_m

Only keep [lon, lat, z] in geometry if:
- the downstream partner explicitly wants it, and
- they confirm their parser uses it correctly.

9. WHY OBJECTS CURRENTLY LOOK FLAT
- Every polygon vertex currently has the same Z value.
- That means the object is represented as one flat polygon at one altitude.
- GeoJSON Polygon does not represent a 3D solid by itself.
- height_m in properties does not automatically create visible height.

10. IMPORTANT IMPLICATION FOR HEIGHT
- The current GeoJSON does not contain volumetric 3D geometry.
- It only contains a footprint (plus optional vertex altitude).
- If the partner wants visible height/extrusion, you need to provide:
  - base_altitude_m
  - height_m
  - optionally top_altitude_m
- Then the downstream system must extrude the footprint.

11. WHAT TO CHECK ABOUT height_m
Action:
- Inspect where height_m is assigned in geojson_bridge.py.
- Figure out whether it is:
  - a placeholder default,
  - a bbox-based estimate,
  - a depth-derived estimate,
  - or just a nominal small value.
- Right now 0.1 looks like metadata only, not a true object-height estimate.

12. PREFERRED OUTPUT MODEL
For semantic mapping:
- Static region-like objects (building, debris, blocked area, vegetation patch, burn area, water, etc.)
  -> Polygon / MultiPolygon
- Thin linear structures (wall, fence, curb, edge)
  -> LineString only if you truly estimate a line
- Small objects with weak depth
  -> fallback proxy polygon or Point, but mark them as approximate

13. ADD GEOMETRY PROVENANCE / QUALITY METADATA
Add fields so downstream users know how reliable the shape is.

Recommended properties to add:
- geometry_source: "mask_projected" | "bbox_proxy" | "point_fallback"
- geometry_confidence
- num_depth_points
- uncertainty_xy_m
- uncertainty_z_m
- track_id
- frame_id
- timestamp
- pixel_bbox

14. KEEP STYLING, BUT SEPARATE IT FROM SEMANTICS
- Keeping stroke / fill / marker-color / marker-size / marker-symbol is fine for visualization.
- But semantic meaning should not depend on the style fields.
- Main semantic fields should stay explicit: class, category, source, detection_type, confidence, etc.

15. MERGED OUTPUT / TRACKING LOGIC
- Since you already track detections over time, the merged GeoJSON should ideally represent persistent semantic objects, not repeated per-frame duplicates.
Action:
- Use track_id / persistent IDs to update and refine existing features across frames.
- Consider keeping 2 outputs:
  1) per-frame detection layer
  2) persistent semantic map layer

17. PRACTICAL DECISION FOR YOUR CURRENT PROJECT
Best current direction:
- For segmentation masks:
  - export Polygon / MultiPolygon based on projected mask geometry.
- For bbox detections:
  - keep bbox in metadata,
  - only generate geographic footprint if depth-based projection supports it.
- Default to 2D [lon, lat] geometry.
- Store altitude and object-height info in properties.

3) Default GeoJSON geometry to 2D [lon, lat].
4) Move altitude and height into properties.
5) Verify whether GNSS height is ellipsoidal or converted to sea-level-like elevation.
6) Investigate why height_m is 0.1 and whether it is only a placeholder.
7) Add provenance / uncertainty fields.
